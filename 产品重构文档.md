## 3. 产品重构总目标

本轮重构的总体目标是：

1. 界面简化
   -  去掉无必要元素、无意义装饰、冗余设置、重复页面和假入口。 
   -  将英文配置项、本地页面文案统一改为中文。 
2. 任务系统收敛
   -  将原有任务体系统一收敛为**一个任务看板**。 
   -  任务看板以“**监控与留痕**”为主，而不是复杂流程编排工具。 
   -  任务查看维度保留为：**看板 + 日程**。 
3. 团队与 Agent 成为核心
   -  强化团队总览、团队地图、员工广场（Agent 广场）能力。 
   -  团队、员工、频道、任务之间形成统一的组织入口。 
   -  支持从团队视角和员工视角进行查看、私聊、记忆管理、技能管理。 
4. 多系统接入
   -  接入**飞书、钉钉、企微、QQ、微信**。 
   -  接入模式限定为**机器人接入**，不接群聊。 
   -  飞书接入必须支持**显式指定负责人/对应 Agent**，避免默认绑定错误对象。 
5. 设置项清理
   -  仅保留真正有价值的设置。 
   -  模型、多模型路由、工具权限、记忆、技能、备份迁移、应用更新、关于等保留。 
   -  无意义、重复、假页面、实验性质页面删除。 

------

## 4. 核心设计原则

### 4.1 极简但不能丢能力

-  去掉多余 UI，不等于削弱核心能力。 
-  要保留真正影响使用的关键能力：文件、Agent、模型、任务监控、团队、技能、工具权限、记忆、接入。 

### 4.2 以团队协作与 Agent 协作为核心

-  用户不是只和一个单体 Agent 交互，而是要管理**团队 + 员工 + Leader + 会话 + 任务**。 
-  产品结构要围绕“团队编排 + 员工培养 + 留痕监控”展开。 

### 4.3 以用户理解成本最低为原则

-  页面名称统一中文化。 
-  页面数量减少。 
-  侧边栏结构更明确。 
-  一个模块只承担一种主要职责。 

### 4.4 监控优先于调度复杂化

-  任务看板的核心是“看”和“留痕”。 
-  是否执行、何时执行、谁执行，可通过会话和指派完成。 
-  看板主要用于全局掌控和历史追踪。

# 6.3 任务看板（核心重构模块）

## 6.3.1 模块定位

任务看板最终定位为：

-  **统一的唯一任务看板** 
-  以**监控、留痕、全局查看**为核心目标 
-  不是复杂执行引擎，也不是重型 workflow 工具 

## 6.3.2 看板结构

统一为一个任务看板，仅保留两个视图：

1.  **看板** 
2.  **日程** 

其他类似：

-  总览 
-  流水线 
-  右侧新建建筑物/冗余面板 

都删除。

------

## 6.3.3 看板状态列

看板中仅保留 4 个状态：

1.  待办 
2.  进行中 
3.  审查 
4.  完成 

删除：

-  积压（与待办重复） 

------

## 6.3.4 状态变更方式

任务状态可通过两种方式改变：

1. 点击操作
   -  如点击“开始处理” 
2. 拖动操作
   -  例如从“待办”拖到“进行中” 

会议对“拖动是否有用”有过讨论，但最终保留“点击或拖动都可以”的能力表达。

------

## 6.3.5 任务创建方式

会议中对“是否保留前台新建任务按钮”有反复讨论，按后文裁决，最终要求如下：

### 最终倾向

-  任务看板主要承担**监控与留痕** 
-  **更推荐从对话中创建任务** 
-  即用户直接在 session / 对话中说明“你要干什么”，由系统创建任务并进入看板留痕 

### 补充说明

-  会议中曾提到“新建任务也能保留”，但后续明确强调“看板最终目的就是看”“只做一个监控功能” 
-  因此，本版产品文档将其定为： 
  -  **对话创建任务：保留且重要** 
  -  **看板前台新建任务按钮：非核心，可弱化或不作为优先实现项** 

------

## 6.3.6 看板展示维度

任务看板需要支持两类监控维度：

### 维度 A：按 Agent / 团队维度监控

-  监控每个 Agent 底下有哪些待办 
-  监控已完成任务 
-  让用户看到每个 Agent 的任务负载和产出情况 

### 维度 B：按时间维度监控

-  从人/团队的时间安排维度查看任务 
-  便于排期 
-  便于把 Agent 的可用时间“花完”，最大化 Agent 成本利用率 

------

## 6.3.7 日程视图

日程视图要求：

-  采用**日历插件** 
-  支持查看周一到周日，以及不同月份和年份
-  支持按时间查看安排 
-  切换不同月份时，仍可看到当月历史事件 
-  **必须保留历史记录** 

------

## 6.3.8 任务归属显示

任务看板中需要区分：

-  团队负责的任务 
-  员工（个人 Agent）负责的任务 

区分方式：

1. 前缀区分
   -  如 team / 团队前缀 
2. 颜色区分
   -  团队与员工显示不同颜色 

### 示例语义

-  团队 N 负责“审查代码” 
-  ZQ 小助手作为员工负责另一件事 

这两类任务都进入同一个统一看板中，但展示上要可区分。

# 6.6 团队系统（Team）重构

## 6.6.1 团队系统定位

会议明确表示：**team 是关键模块**。

团队系统承担：

-  团队组织查看 
-  团队创建与解散 
-  leader 与成员编排 
-  团队地图 
-  成员 skills 管理 
-  memory 管理 
-  私聊 / Chat 
-  团队与个人任务关系查看 

------

## 6.6.2 删除的团队相关页面

以下页面删除：

-  团队架构 
-  角色策略 
-  通道高级 
-  自动化默认策略 
-  相关“工作流那 3 个”冗余页面 

### 原因

-  会议认为这些内容没用、重复或不符合当前产品组织方式 
-  当前产品组织方式更接近： 
  -  老板 
  -  经理 
  -  员工 
-  即存在天然层级，不需要单独做复杂团队架构页 

------

## 6.6.3 团队总览

团队总览是团队模块的主入口之一。

### 展示内容

团队总览中展示：

-  team 名称 
-  team 职责 
-  team leader 
-  团队状态 
-  最近活跃 
-  团队基础信息 

### 不展示

-  模型信息不放在团队总览里 

### 操作能力

-  解散团队 
-  新加团队 

------

## 6.6.4 新建团队

点击“新加团队”后，进入团队创建流程。

### 创建方式

界面结构包括：

-  两条类似流水线的区域 
-  右侧 Agent 列表 

### 拖拽规则

-  可从 Agent 列表中拖拽 Agent 
-  leader 区只能放 **1 个** 
-  其余成员拖入“团队成员”区域 

### 数量策略

-  不做严格硬限制 
-  但建议团队成员数控制在 **2~3 个**（会议最终建议） 
-  相关考虑与 free memory / 管理成本有关 

### 创建后的联动更新

团队建立后，需要更新：

-  成员之间关系 
-  长期 Memory 
-  soul 
-  identity 
-  引用关系 

会议中明确提出：

-  团队建立后相关信息会更新 
-  成员之间会互相有引用关系 

------

## 6.6.5 团队卡片与进入团队地图

### 团队卡片

-  团队列表以卡片方式展示 
-  可参考开源项目卡片式呈现 
-  可更直观展示成员数量、团队规模等 
-  例如 6 人团队视觉上要直观、好看

------

## 6.6.6 团队地图

团队地图是团队内部管理页面。

### 能力要求

1.  查看团队成员信息 
2.  管理员工 skills 
3.  删除员工 
4.  增加员工 
5.  右键私聊 / Chat 
6.  修改 memory 

### 增加员工

-  可在团队地图中新增员工 
-  点击后再次出现 Agent 列表进行选择 
-  属于可编辑式团队管理 

### Skills 管理

-  可查看和编辑每个员工的 skills 

### Memory 管理

会议对“memory 是否能在这里改”有前后变化，最终以后文为准：

**最终结论：可以在团队地图里直接改 memory。**

并且：

-  修改后要与后面的记忆知识库同步 
-  实现前台直接点改、逐条修改的体验 

### 展示信息

团队地图中至少可展示：

-  当前任务 
-  待命 
-  活跃 
-  以及当前已有展示信息 

会议结论是：先按当前已有信息展示，再逐步完善。

# 6.7 员工广场（Agent 广场）

## 6.7.1 模块定位

员工广场是 Agent 的展示、创建、培养、管理入口。

------

## 6.7.2 能力要求

### 默认能力

-  默认提供多个模板 

### 新增能力

-  用户可新加 Agent 

### 新增后的行为

-  Agent 加入后即可开始聊天 
-  一旦聊天就会形成 session 
-  通过聊天对 Agent 进行“培养” 

------

## 6.7.3 聊天与会话管理

### 进入聊天方式

-  右键直接私聊 
-  或提供 Chat 按钮 

### 会话结果

-  只要有聊天记录，就会进入左侧会话列表 
-  会话记录支持： 
  -  改名 
  -  删除 
  -  重命名 
  -  设置等管理操作 

------

## 6.7.4 Agent 的 memory 管理

员工广场中，最好直接支持 Agent 的 memory 管理。

### 目的

-  用户可以直接导入优秀 Agent 的设定内容 
-  例如从外部平台（如推特）复制优秀方案后直接粘贴进来 
-  也可以通过聊天慢慢培养，然后保存 

### 数据落点

-  memory 修改最终要进入该 Agent 对应的 Workspace / 知识库区域 
-  与记忆知识库保持同步 

------

## 6.7.5 员工广场的视觉风格

## 9. 关键交互说明

# 9.1 从对话创建任务

用户在会话中直接说：

-  “帮我做……”

-  “安排……去执行……”

2.  进入任务看板 
3.  在看板中留痕 
4.  可再通过点击/拖动切换状态

# 9.2 创建团队

用户点击“新加团队”后：

1.  右侧展示 Agent 列表 
2.  拖 1 个 Agent 到 leader 区 
3.  拖若干 Agent 到成员区 
4.  建立团队后，自动更新团队内部引用与长期记忆关系 
5.  团队卡片出现在团队总览中 
6.  点击卡片进入团队地图

## 11. 需特别注意的实现联动

以下联动在本轮设计中非常关键：

1. 模型配置 ↔ 模型提供商
   -  前后台配置一致 
   -  支持多模型 
   -  支持优先级与路由 
2. 团队创建 ↔ memory / soul / identity / 引用关系
   -  团队建立后需联动更新 
3. 团队地图 ↔ 记忆知识库
   -  memory 在团队地图可改 
   -  修改后需同步知识库 
4. 聊天 ↔ 会话 ↔ memory
   -  聊天可形成会话 
   -  聊天本质上也是 memory 培养方式 
5. 任务看板 ↔ 团队/员工
   -  统一看板中展示不同归属任务 
   -  通过前缀和颜色区分 
6. 外部接入 ↔ Leader / Agent
   -  飞书等接入必须显式指定绑定对象 
   -  不允许默认错绑
