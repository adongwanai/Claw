最后冲刺：布局居中、彻底解决拖拽性能、以及跨团队的 Leader 后缀逻辑

目前的 UI 已经比最初好多了，但还存在三个严重影响体验的硬伤。请立即按以下要求进行精准修复：

1. 视觉布局对齐：解决“不居中”的问题

问题：在宽屏下，顶部的“新建团队”拖拽接收区以及下方的“团队卡片网格”都紧贴着左侧，右侧留有大量空白，视觉极度不平衡。

修复方案：

请给主内容区（包含拖拽区和卡片网格的容器）加上最大宽度和居中对齐，例如：max-w-6xl mx-auto w-full。

确保拖拽接收区（Dropzone）在这个居中容器内占满宽度或美观地居中展示。

团队卡片网格（Grid）也需要在这个容器内均匀分布，不要全部挤在左边。

2. 业务逻辑补充：跨团队的 Leader 与 Team Name 后缀

当前错误行为：如果我已经用 Agent "Main" 创建了一个名为“Main 的团队”，当我再次用 "Main" 创建新团队时，默认名字还是“Main 的团队”，Leader 也叫 "Main"，导致完全无法区分。

修复方案（全局重复检测）：

当用户把 Agent 拖入 Leader 区时，请遍历系统中已有的团队。

如果发现该 Agent 已经作为 Leader 存在于其他团队中，请自动累加后缀。

团队默认名称应该变为：[Agent名称] 的团队-[N]（例如：Main 的团队-1, Main 的团队-2）。

Leader 的展示别名也应该相应变为：[Agent名称]-[N]（例如：Main-1, Main-2）。确保这个后缀逻辑在生成确认弹窗前就已生效。

3. 终极性能优化：彻底终结“拖拽卡顿”
如果现在还卡，说明你的拖拽库实现依然存在严重的“渲染雪崩（Re-render avalanche）”。请从代码底层进行以下强制优化：

使用 React.memo：给右侧面板中的 AgentCard 组件、左侧的 Dropzone 组件全部套上 React.memo，防止父组件状态更新时它们被无辜连累重绘。

清空 onDragMove：如果你使用了 @dnd-kit，请确保 onDragMove 事件中绝对没有调用任何 setState（除非是为了极其必要的视觉反馈）。复杂的碰撞检测和数组更新只能放在 onDragEnd 中执行。

DragOverlay 检查：确保你渲染的 DragOverlay（跟随鼠标的拖拽替身）是一个纯展示的极简 DOM，不携带复杂的响应式状态或庞大的子组件。

传感器优化 (Sensors)：为拖拽传感器增加一点容差值（例如 PointerSensor 设置 activationConstraint: { distance: 5 }），避免鼠标微小抖动就触发复杂的拖拽计算逻辑。

请务必先集中精力把拖拽流畅度做到丝滑，再处理居中和后缀逻辑。不流畅的拖拽是不可接受的。
